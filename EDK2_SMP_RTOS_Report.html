Project Overview

This report documents the complete development process of a multi-CPU RTOS running on EDK2 UEFI firmware in QEMU ARM64 environment. The system demonstrates parallel execution of tasks across 4 Cortex-A53 CPUs using EFI_MP_SERVICES_PROTOCOL.

Key Components:

  • Firmware: EDK2 ArmVirtQemu (AARCH64 RELEASE)
  • Application: SmpRtos.efi - Custom UEFI application
  • Platform: QEMU ARM Virt machine with 4 CPUs
  • Protocol: EFI_MP_SERVICES_PROTOCOL for multi-CPU coordination

Phase 3: Bug Identification and Resolution

Bug 1: RNG Library Crash (RNDR Instruction)

Symptom: EDK2 firmware crashed with Synchronous Exception at early boot

Root Cause: RngDxe driver uses ARMv8.5 RNDR instruction not supported by QEMU 4.2

Synchronous Exception at 0x5F7ACECC ESR: EC 0x00 IL 0x1 ISS 0x00000000

Fix 1: Replace RNG Library

Patched ArmVirtQemu.dsc to use BaseRngLibNull instead of BaseRngLib

# In ArmVirt.dsc.inc
RngLib|RngLibNull.inf

# In ArmVirtQemu.dsc
# Comment out:
# RngDxe

Bug 2: Missing MP Core Info HOB

Symptom: MpServiceProtocol not found after firmware boot

Root Cause: ArmPsciMpServicesDxe requires gArmMpCoreInfoGuid HOB to enumerate CPUs

ERROR: Cannot find MpServiceProtocol: EFI_NOT_FOUND

Fix 2: Create MP Core Info PEIM

Created custom PEIM to publish HOB with 4 CPU entries for QEMU virt platform

/* ArmVirtQemuMpCoreInfoPeim.c */
#define QEMU_CPU_COUNT 4
ARM_CORE_INFO CoreInfo[QEMU_CPU_COUNT];
for (i = 0; i < QEMU_CPU_COUNT; i++) {
CoreInfo[i].Mpidr = (UINT64)i;
CoreInfo[i].Mailbox* = 0; // PSCI handles wakeup
}
BuildGuidDataHob(&gArmMpCoreInfoGuid, CoreInfo, sizeof(CoreInfo));

Added to DSC and FDF files to include in firmware build.

Bug 3: PSCI Conduit Mismatch

Symptom: Synchronous Exception when AP tries to execute after StartupThisAP

Root Cause: ArmPsciMpServicesDxe uses ArmCallSmc (smc #0) but QEMU virt PSCI uses HVC

// QEMU device tree shows:
psci {
method = "hvc";
compatible = "arm,psci-0.2";
}

Fix 3: Custom ArmSmcLib with HVC

Created SmpRtosPkg/Library/ArmSmcLibHvc that overrides ArmCallSmc to use hvc #0

// AArch64/ArmSmc.S
ASM_FUNC(ArmCallSmc)
// ... load args ...
hvc #0 // Changed from smc #0
// ... store results ...

Override in DSC for ArmPsciMpServicesDxe only:

ArmPkg/Drivers/ArmPsciMpServicesDxe/ArmPsciMpServicesDxe.inf {
<LibraryClasses>
ArmSmcLib|SmpRtosPkg/Library/ArmSmcLibHvc/ArmSmcLibHvc.inf
}

Step 4.2: Implement Multi-CPU Task Execution

Used StartupAllAPs with blocking mode for true parallel execution

Status = MpServices->StartupAllAPs(
MpServices,
CpuTask, // Task function
FALSE, // SingleThread = FALSE (parallel)
NULL, // WaitEvent = NULL (blocking)
0, // Timeout = 0 (infinite)
(VOID *)&Shared, // Shared context
NULL // FailedCpuList
);

Step 4.3: CPU Identification in Tasks

Each AP calls WhoAmI to identify itself since StartupAllAPs shares one context

STATIC VOID EFIAPI CpuTask (VOID *Context)
{
SHARED_CONTEXT *Shared = (SHARED_CONTEXT *)Context;
UINTN Cpu = 0;
Shared->MpServices->WhoAmI(Shared->MpServices, &Cpu);
// ... execute tasks ...
}

Key Technical Insights

1. EDK2 Build System

  • Cross-compilation requires GCC_AARCH64_PREFIX environment variable
  • Library overrides can be applied per-module in DSC files
  • PEIMs execute in PEI phase before DXE drivers

2. Multi-CPU Boot Sequence

  1. BSP (CPU 0) boots EDK2 firmware
  2. PEIM publishes MP Core Info HOB
  3. ArmPsciMpServicesDxe loads and enumerates CPUs via HOB
  4. UEFI application locates MpServiceProtocol
  5. StartupAllAPs launches tasks on all APs simultaneously
  6. BSP blocks until all APs complete, then runs its own tasks

3. QEMU Compatibility Issues

  • QEMU 4.2 uses HVC for PSCI, not SMC (ARM standard)
  • ARMv8.5 instructions (RNDR) not emulated in older QEMU
  • Device tree must be inspected to understand platform configuration

Lessons Learned

  • Always check platform device tree - QEMU's PSCI method differs from hardware
  • Use blocking StartupAllAPs - Non-blocking mode with spin-wait can deadlock
  • HOBs are critical for MP services - Without proper HOB, drivers won't initialize
  • Cross-compiler environment - EDK2 build system is sensitive to toolchain setup
  • Incremental testing - Build firmware, test, then build application separately
